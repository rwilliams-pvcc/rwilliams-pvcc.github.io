Software Engineering

TERM DEFINITION
software engineering A branch of computer science that deals with the design, development, testing, and maintenance of software applications.
integrated development environment (IDE) A system that combines tools for developing softwares (editors, compilers, debugging tools, and so on) into a single, integrated package. An example for these systems are those developing applications for smartphones.

Software Life Cycle: Development phase
  • Requirements analysis
  • Design
  • Implementation
  • Testing
  • The goal of which is to specify what services the proposed system will provide, to identify any conditions (time constraints, security, and so on) on those services, and to define how the outside world will interact with the system.
  • Involves creating a plan for the construction of the proposed system. In a sense, design is about developing a solution to the problem, the problem being requirements analysis.
  • Involves the actual writing of programs, creation of data files, and development of databases. At this stage, we see the distinction between the tasks of a software analyst and a programmer.
  • In traditional development, testing was essentially equated with the process of debugging programs and confirming that the final software product was compatible with the software requirements specification. Today that has changed. Testing is seen as part of the process of software development instead of being a separate step.
modular implementation The division of software into manageable units, generically called modules, each of which deals with only a part of the software's overall responsibility.
Unified Modeling Language (UML) A modern collection of tools that has been developed with the object-oriented paradigm in mind.
use case The interactions in a use case diagram.
use case diagram Depicts the proposed system as a large rectangle in which interactions (called use cases) between the system and its users are represented as ovals and users of the system (called actors) are represented as stick figures (even though an actor may not be a person).
Data relationships
  • one-to-one
  • one-to-many
  • many-to-many
  • Exemplified by the association between patients and occupied private rooms in that each patient is associated with only one room and each room is associated with only one patient.
  • Exemplified by the association between physicians and patients in that one physician is associated with many patients and each patient is associated with one (primary) physician.
  • Would occur if we included consulting physicians in the physician-patient relationship. Then each physician could be associated with many patients and each patient could be associated with many physicians.
design pattern A predeveloped model for solving a recurring problem in software design.
Software quality assurance (SQA) Groups that are developed and charged with overseeing and enforcing the quality control systems adopted by the organizations.
Software testing:
  • glass box testing
  • black box testing
  • beta testing
  • alpha testing
  • The software tester is aware of the interior structure of the software and uses this knowledge when designing the test.
  • Refers to tests that do not rely on knowledge of the software's interior composition. Performed from the user's point of view and not concerned with how the software goes about its task but merely with whetehr the software performs correctly in terms of accuracy and timeliness.
  • A preliminary version of the software is given to a segment of the intended audience iwht the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market.
  • Similar to beta testing, however testing is performed at the developer's site.
Documentation
  • user documentation
  • system documentation
  • technical documentation
  • The purpose is to explain the features of the software and describe how to use them. It is intended to be read by the user of the software and is therefore expressed in the terminology of the application.
  • The purpose is to describe the software's internal composition so that the software can be maintained later in its life cycle.
  • The purpose is to describe how a software system should be installed and serviced (such as adjusting operating parameters, installing updates, and reporting problems abck to the software's developer).
Intellectual property law
  • software license
  • software disclaimer
  • best defenses against software liability claims
    Legal efforts to provide ownership fall under this category of law, which much of which is based on the well-established principles of copyright and patent law.
  • The rights of the developer are formally expressed in legal terms in a software license.
  • Software developers often include disclaimers in the software licenses that state the limitations of their liability.
  • The best defenses against software liability claims is to apply sound software engineering principles during the software's development, to use a level of care compatible with the software's application, and to produce and maintain records that validate these endeavors.
